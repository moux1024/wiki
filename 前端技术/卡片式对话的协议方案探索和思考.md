# 卡片式对话的协议方案探索和思考

> 日期: 2026-04-17

本文探讨了智能助手对话流中卡片式交互的系统性工程方案，围绕三个核心问题展开系统性思考。

## 核心问题一：卡片如何嵌入对话流

### 方案 A：代码块扩展（✅ 生产采用）

借用 Markdown 代码块的 language 字段作为组件类型标识：

````
```card
{ "type": "weather", "data": {...} }
```
````

- 优势：天然兼容 Markdown 解析器、流式友好（代码块天然支持增量拼接）、前端只需识别特定 language 即可替换渲染
- 劣势：数据格式需自行约定

### 方案 B：占位符替换

模型输出特殊标记（如 `{{card:weather}}`），前端识别后替换为组件。

- 劣势：流式场景下体验差——用户会先看到占位符闪烁再变成卡片

### 方案 C：自定义标签（XML-like）

灵感来自 bolt.new，使用 XML 风格标签：

```xml
<card type="weather" city="杭州" />
```

- 优势：表达力最强，可嵌套、可携带属性
- 劣势：需要独立的 XML parser，与 Markdown 混排有复杂度

## 核心问题二：卡片数据从何而来

### 演进路径

**阶段 1：模型直出**
- 模型直接在文本中输出 JSON 数据
- 适合静态内容（如代码片段、格式化文本）
- 不适合实时业务数据（如库存、订单状态）

**阶段 2：增量 Patch 更新**
- 模型输出骨架 ID，前端先渲染骨架屏
- 服务端异步获取真实数据后，通过 Patch 替换骨架
- 提升首屏体验，但增加架构复杂度

**阶段 3：Tool 驱动（当前最佳实践）**
- Tool 返回结构化数据 + UI 描述
- 前端按约定渲染，无需硬编码
- 两种主流协议：
  - **MCP Apps**：工具驱动 UI，Tool 定义返回类型中包含 UI Schema
  - **A2UI（Agent to UI）**：Agent 驱动 UI，Agent 自主决定何时渲染什么 UI

## 核心问题三：四层统一协议体系

| 层级 | 名称 | 职责 | 示例 |
|------|------|------|------|
| 第一层 | Markdown 标记协议 | 卡片在文本流中怎么写 | 代码块扩展、自定义标签 |
| 第二层 | 消息传输协议 | 前后端传什么格式 | SSE、WebSocket、事件流 |
| 第三层 | UI 渲染协议 | 卡片长什么样 | A2UI JSON Schema、React Component |
| 第四层 | 事件通信协议 | 用户交互后怎么办 | 回调事件、Action 触发 |

### 关键社区协议

- **AG-UI**：Agent-UI 通信协议
- **MCP**（Model Context Protocol）：模型上下文协议，Anthropic 主导
- **A2A**（Agent to Agent）：Google 主导的 Agent 间通信协议
- **A2UI**（Agent to UI）：Agent 驱动 UI 渲染协议
- **MCP Apps**：MCP 扩展，工具返回 UI Schema
- **Open-JSON-UI**：开源 UI Schema 标准
- **Web MCP**：浏览器端 MCP 实现

## 实践建议

1. **优先采用代码块扩展方案**嵌入对话流，兼容性和流式支持最好
2. **数据获取走 Tool 驱动**，结合 MCP Apps 或 A2UI 协议
3. **四层协议分层设计**，各层可独立演进
4. **关注社区协议发展**，避免重复造轮子
