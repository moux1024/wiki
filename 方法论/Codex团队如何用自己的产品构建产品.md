## 核心观点

Codex 团队几乎不写产品规格文档，整个产品的 spec 只有 10 个要点。50-100 人的团队长期只有一个 PM，设计师写的代码量超过了六个月前工程师写的代码量。

## 极简文档哲学

- 只有问题复杂到一人装不下、需多人协调时才写 spec
- 即使写，也只有十来个要点
- **核心原则**："让离金属最近的人做尽可能多的决策"
- 大部分编码工作可委派给 AI Agent，一人能处理的范围大了很多

## PM 的三种 Codex 使用模式

1. **简单改动**：直接用 Codex 生成代码、测试、提 PR。提一个好 PR 比找人沟通排优先级更快
2. **中等复杂度**：让 Codex 先做实现计划
3. **模糊想法**：和模型对话探索代码库，建立心智模型，把理解（非计划本身）分享给工程师

> "我不是在写代码，我是在建立心智模型。"

## 设计师角色的变化

Codex 团队的设计师现在写的代码量，超过了六个月前一个工程师写的代码量。关键问题不再是"你能不能生成代码"，而是"你选择做的事情对不对"和"怎么保证质量"。

团队在**系统思考和质量把控**上投入大量精力，不是凭感觉编程（vibe coding）。

## 规划方法：只做近期和远期

OpenAI 内部的规划哲学：

- **近期**：8 周以内，具体目标，团队集中冲刺
- **远期**：方向感（"vibes"），如一年后更聪明的模型、多 Agent 并行、用户无需主动提示
- **中期路线图**：基本不存在。模型能力快速变化，中期规划变成猜测

## Codex App 的诞生

远期愿景是让多 Agent 协作自然化，不绑定特定文件夹或 IDE。

关键转折：2025 年 12 月 GPT-5.2 Codex 发布，模型能力跨过"可以可靠长时间独立工作"的门槛。用户开始用 tmux 同时运行大量 Codex 会话。社交媒体上有人在三块显示器上开了约 18 个终端窗口。

但只有顶尖 1% 的工程师会用终端并行方式工作。团队内部争论后决定做独立桌面应用。推动项目启动的不是 spec，而是一份"为什么我们认为做一个 app 是好主意"的文档。

技术细节：Codex App、IDE 扩展和 CLI 底层共用同一套开源的 Rust 核心（Codex harness）。

Codex App 于 2026 年 2 月正式发布，发布当月超 100 万开发者使用。

## 产品设计原则

**先为 power user 做可配置性，再为普通用户做简化。**

- 核心交互极简——"让产品几乎隐形，让模型的能力自然显现"
- 通过分层让 power user 自己发现更深功能
- 比喻：用 Codex App 像打游戏，一层层解锁

> "最前沿的用户和我们一起住在未来，还把我们往未来拽。"

## 海盗船式团队运作

- 50-100 人团队长期只有一个 PM，直到最近才有第二个
- 跨职能协调极少，汇报结构极简
- **有意识地像一支海盗船**

挑战：越来越多 OpenAI 内部非技术团队也在用 Codex App 做非编码工作，定位从"开发者工具"向更通用方向扩展。

## 开发者体验战略

Codex 定位为 OpenAI 整个开发者平台的入口，不只是编码。

新策略：直接教开发者用 Codex + Skill 来完成集成（Imagen、Sora、Speech-to-Speech 等），而非手动撰写指南。

社区建设已在多个城市建立 Codex 大使网络。

## PM 是填空岗位，不是领导岗位

Alex 的核心判断：

> "我不认为 PM 是一个好的领导岗位。它是一个填补空缺的岗位。"

推理链条：AI 工具让每个人能做一部分别人的工作 → 工程师有了更多时间 → "人才栈压缩"正在发生 → 做事需要的人越少，决策越纯粹。

**细分**：
- 想做工程师却因管理能力做了 PM → 直接转工程师
- 更喜欢设计 → 去做设计
- 真正感兴趣花时间和用户在一起、理解市场 → PM 可能还有空间

> "每个问题都需要一个负责的人，但这个人不一定得是 PM。"

对照案例：**Stripe 在没有任何 AI 工具的情况下，250 名员工零 PM**。前提条件：他们全是工程师，构建自己一直想要的 API，每个人都知道优雅的 API 长什么样。

**PM 需求的大小和你离用户的距离有关。**

## 招聘标准

一个词：**能动性（agency）**。

- 入职没有递增难度任务列表，就是"欢迎，自己找事做"
- 看链接和想法，不看自我介绍和简历
- 不知道团队成员是什么学校毕业的

> "我很高兴我们生活在一个那些愚蠢的证书不再重要的世界里。管它什么学校呢。给我看你做了什么。"

## 结构性前提（值得反思）

Codex 团队的极简管理方式建立在一个特殊条件上：

1. **他们是自己产品的用户**
2. **他们的用户也是开发者**
3. **有极度活跃的开源社区提供持续反馈**

当 Codex 的野心是"把编码 Agent 带给 ChatGPT 的 9 亿用户"时，这套方法还能继续有效吗？

> 当一个编码 Agent 产品的用户从"写代码的人"扩展到"所有人"，产品决策的复杂度会发生质变。
