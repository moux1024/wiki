# 6个让Agent技能稳定生效的模式，适用于Claude Code、Codex

大多数人写的 Claude Skill 从来不触发，或者触发了也输出乱七八糟。

不是 Claude 不够聪明，是技能文件本身写错了。

@zodchiii 整理了六个让技能真正生效的模式，以下是完整拆解。

## 先说清楚 Skill 是什么

Skill 是一个带 YAML frontmatter 的 Markdown 文件，放在 `~/.claude/skills/skill-name/SKILL.md`。

当你的描述与当前上下文匹配时，Claude 会自动加载它；也可以用 `/skill-name` 手动触发。

结构本身没什么难度，难的是 `SKILL.md` 里面写什么。

## 模式一：描述要告诉 Claude「什么时候用」，不只是「用来干什么」

这是最重要的字段。Claude 在决定加载哪个技能之前，会扫描所有可用技能的描述。如果描述只说功能，Claude 不知道什么时候该调用它。

**不好的描述：**
`"生成 commit message"`

**好的描述：**
`"当用户要提交代码、输入 commit 或问如何写提交信息时，生成符合约定式提交规范的 commit message，关键词：git commit、提交、提交信息"`

好的描述同时告诉 Claude 三件事：技能做什么、何时触发、监听哪些关键词。

描述不足 50 个字的技能，被调用的概率比有完整触发上下文的技能低 3 到 5 倍。而且只有描述的前 250 个字符会被纳入 Claude 的上下文预算，所以要把最关键的触发条件放在前面。

## 模式二：用指令语气，不要用对话语气

技能是指令文档，不是聊天记录。

**弱（对话式）：**
`"你可能需要考虑先看一下现有的文件结构"`

**强（指令式）：**
`"读取当前目录下的测试文件。按照相同的命名规范，为目标函数生成测试用例。输出格式：每个用例独立一个代码块，附上中文注释说明测试意图。"`

Claude 对祈使句的响应要稳定得多。Anthropic 官方仓库里的每一个技能，都遵循同一个节奏：直接动词、编号步骤、明确的输出格式。

## 模式三：指定输出格式

这是大多数自定义技能失败的地方。只说要做什么，不说输出长什么样，Claude 每次会自己发明格式，结果不一致。

**不指定格式：**
你得到的可能是一行，可能是几段，可能带前缀，可能不带。

**指定格式之后：**
每次得到同样的输出结构，技能才能被稳定复用。

格式说明不需要复杂，比如："输出三个备选标题，每行一个，不加编号，不加解释" 就够了。

## 模式四：加入「先读」步骤

好的技能不假设 Claude 了解你的项目，而是明确告诉它先读再做。

以 awesome-claude-skills 里的 `/test` 技能为例，它不是直接写「生成测试」，而是先说：读取现有测试文件，理解命名规范和断言风格，再按照相同的模式生成新测试。

这一步的区别在于：不加这步，Claude 给你通用代码，跑起来可能报错；加了这步，Claude 给你跟项目吻合的代码。

## 模式五：明确写出这个技能「不做什么」

反直觉，但很有效。最好的技能会显式声明超出范围的事项。

Anthropic 官方 PDF 技能里有这样一段：

> 本技能不转换成 Word 格式，不直接编辑 PDF，不处理加密保护的文件。

这样写的好处是：当用户提出超出范围的要求，Claude 不会硬着头皮去做然后失败，而是选择另一个合适的技能，或者直接询问澄清。

高质量技能中有约 70% 包含这个模式，低质量技能里几乎没有。

## 模式六：控制在 500 行以内

每次技能被调用，整个文件都会加载进 Claude 的上下文。一个 2000 行的技能在做任何事之前就先消耗了 5000+ 个 token。

文件越长，Claude 越容易在中途失去焦点，开始忽略底部的指令。

Anthropic 官方技能（frontend-design、code-review、security-guidance）都在 300 行以内，安装量最高的社区技能也类似。

如果技能文件开始变长，把它拆开。用「渐进披露」的方式：主文件 `SKILL.md` 里只引用其他文件路径，Claude 只在任务真正需要时才加载那些支撑文件。

## 真实案例：/commit 技能的六条全部对

一个好的 `/commit` 技能是这样的：

- 描述告诉 Claude 何时触发（「用户要提交代码时」）
- 指令是祈使句加编号步骤
- 输出格式明确（提交信息的结构）
- 内置「先读」步骤（先运行 git status、git diff）
- 明确定义不做什么（不推送，不修改提交历史）
- 总长度不超过 50 行

这个技能在任何项目、任何语言下都能稳定工作。

## 哪些写法会杀死技能

失败的技能几乎都有这些特征：

- 描述不足 50 个字
- 没有输出格式
- 用对话语气而不是指令语气
- 没有「先读」步骤
- 没有边界声明
- 超过 500 行

## 怎么修已有的技能

找出你最弱的那个技能（就是 Claude 从来不主动调用那个），对着上面的清单逐条检查，修完之后，直接跟 Claude 说一句它应该会用这个技能的话，但不要手动触发。

如果 Claude 自动选择了正确技能，描述起效了。如果它选了别的，或者干脆什么都没选，描述里需要补充更多触发上下文。

## 技能的复利效应

写得差的技能有三个代价：多耗上下文、输出不稳定、路由出错。

写得好的技能反过来叠加。每一个设计良好的技能，都能让 Claude 更准确地在整个集合里找到最合适的那个，技能变成了系统，而不只是文件的堆叠。

用了六个月的人的技能文件夹，和刚开始用的人完全不同。Claude 是同一个，但输出质量已经不在一个量级。
