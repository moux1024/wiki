# LLM Wiki — 用 LLM 构建个人知识库的模式

> 来源: https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
> 日期: 2026-04-08
> 分类: AI
> 标签: LLM, 知识管理, RAG, Agent, Obsidian

## 核心思想

大多数人对 LLM 和文档的使用方式是 RAG：上传文件集合，LLM 在查询时检索相关片段生成答案。这种方式的问题是**没有知识积累**——每次提问，LLM 都要从头重新发现知识。

本文提出一种不同的方式：LLM **增量构建和维护一个持久化的 wiki**——一组结构化的、相互关联的 markdown 文件。当你添加新来源时，LLM 不仅索引它，还会提取关键信息并整合到现有 wiki 中——更新实体页面、修订主题摘要、标注新旧数据的矛盾、强化或挑战不断演进的综合性观点。知识只编译一次，然后**持续保持更新**。

**关键区别：wiki 是一个持久的、复利增长的产物。** 交叉引用已经建立，矛盾已标记，综合分析已反映所有已读内容。每添加一个来源、每问一个问题，wiki 都变得更丰富。

## 适用场景

- **个人成长**：追踪目标、健康、心理、自我提升——整理日志、文章、播客笔记
- **研究**：深入某个主题——阅读论文、文章、报告，增量构建综合性 wiki
- **读书**：每章存档，构建角色、主题、情节页面，最终形成丰富的伴侣 wiki
- **商业/团队**：内部 wiki，由 LLM 维护，输入来自 Slack、会议记录、项目文档、客户通话
- **竞品分析、尽职调查、旅行规划、课程笔记、爱好深潜**

## 架构三层

1. **原始来源（Raw Sources）**——你收集的原始文档（文章、论文、图片、数据文件）。不可变，LLM 只读不改。
2. **Wiki**——LLM 生成的 markdown 文件目录。摘要页、实体页、概念页、比较分析、概述、综合。LLM 完全拥有这一层。
3. **Schema（模式文档）**——如 CLAUDE.md 或 AGENTS.md，告诉 LLM wiki 的结构、约定和工作流。这是关键配置文件。

## 核心操作

### 摄入（Ingest）

放入新来源 → LLM 阅读 → 与你讨论要点 → 写摘要页 → 更新索引 → 更新相关实体/概念页 → 追加日志。单个来源可能触及 10-15 个 wiki 页面。

### 查询（Query）

向 wiki 提问 → LLM 搜索相关页面 → 综合回答并引用。**重要洞察：好的答案可以作为新页面存回 wiki。** 你的探索像摄入的来源一样在知识库中复利积累。

### 检查（Lint）

定期让 LLM 健康检查：页面间矛盾、过时声明、孤立页面、缺失的交叉引用、数据空白。LLM 擅长建议新问题和新来源。

## 索引与日志

- **index.md**（内容导向）：wiki 中所有页面的目录，按类别组织，含链接和一行摘要。LLM 每次摄入时更新。回答查询时先读索引再深入。
- **log.md**（时间导向）：追加式记录，格式如 `## [2026-04-02] ingest | 文章标题`。可用 `grep "^## \[" log.md | tail -5` 快速查看最近条目。

## 可选工具

- **[qmd](https://github.com/tobi/qmd)**：本地 markdown 搜索引擎，混合 BM25/向量搜索 + LLM 重排序，支持 CLI 和 MCP server
- **Obsidian Web Clipper**：浏览器扩展，将网页文章转为 markdown
- **Marp**：基于 markdown 的幻灯片格式，Obsidian 有插件支持
- **Dataview**：Obsidian 插件，对页面 frontmatter 运行查询

## 为什么有效

维护知识库的繁琐部分不是阅读或思考——而是**记账工作**：更新交叉引用、保持摘要最新、标注矛盾。人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会厌倦，不会忘记更新交叉引用，一次能触及 15 个文件。

**人类的职责**：策展来源、指导分析、提出好问题、思考意义。
**LLM 的职责**：其他一切。

这个想法在精神上与 Vannevar Bush 1945 年的 Memex 相似——一个私人的、精心策展的知识存储，文档之间的关联与文档本身一样有价值。Bush 无法解决的是**谁来维护**——LLM 解决了这个问题。
